# 下一步清單

更新日期：2026/05/26

這份文件只放最近最重要的下一步，避免 Codex 每次從長對話裡重新判斷優先順序。

## 最高優先

### 1. 完成 Supabase 資料同步

目的：

- 家裡與公司電腦可以共用同一份資料。
- 外部 Chrome、Codex 內建瀏覽器可以看到同一份資料。
- 未來員工端與會員端可以串接同一份資料。

需要：

- Supabase Project URL。
- Supabase anon public key。
- 執行 `supabase-schema.sql`。
- 將 `js/app.js` 的展示模式切換為 Supabase 模式。

### 2. 穩定預約行事曆

要持續檢查：

- 過去日期不可新增預約。
- 同一教練同一時段不可重複預約。
- 同時段一般預約最多 3 組。
- 小班團課獨立容量。
- 拖曳小卡失敗時只用右下角提示。
- 分期與續約章判定正確。

### 3. 穩定會員與票券資料

要持續檢查：

- 票券分批顯示，不要混在一起。
- 贈送堂數獨立顯示。
- 分期堂數與金額正確。
- 退費與儲值金邏輯正確。
- 新朋友完成教練課後可升級為會員。

## 中期目標

### 4. 員工手機端

目的：

- 教練可以看自己的課表。
- 教練可以協助簽到。
- 櫃台可以快速查看今日課程。

注意：

- 不要直接把後台塞進手機版。
- 手機版要另做簡化介面。

### 5. 會員端

目的：

- 會員可以查看自己的預約。
- 會員可以查看剩餘票券。
- 會員可以簽到或掃 QR。

注意：

- 會員端不能看到其他會員資料。
- 需要登入與權限控管。

## 先不要急著做

- 發票完整串接。
- 合約電子簽署。
- 複雜薪資獎金自動化。
- 太細的報表。
- 大型重構。

這些可以等預約、會員、票券與資料同步穩定後再做。

## 近期介面整理

### 4. 統一後台列表規格

目前方向：

- 會員列表、課程及票券、教練管理都採用一致的列表閱讀方式。
- 重要操作放左側。
- 詳細資料用彈跳視窗呈現。

下一步：

- 確認會員列表、課程列表、票券列表、教練列表的欄位命名是否一致。
- 確認所有「新增 / 編輯 / 查看」都能用同一套彈跳視窗邏輯。
- 檢查窄版桌機畫面是否仍好讀。

### 5. 降低主程式維護壓力

目前 `js/app.js` 已經很大，後續改功能時容易互相影響。

建議：

- 先不要大拆檔案。
- 每次只整理一個區塊，例如「課程管理」、「教練管理」、「會員票券」。
- 等資料同步穩定後，再考慮把會員、預約、課程、教練邏輯分成不同檔案。
## 2026/05/25 下一步：前台 / 教練端串接

等使用者之後輸入「串接」時，再開始執行正式串接，不需要現在睡前繼續動程式。

1. 先讓太太閱讀 `docs/FRONTEND_INTEGRATION.md`，確認會員端與教練端第一版範圍。
2. 決定登入方式：
   - 會員端：先用手機號碼測試，之後再接 LINE / OTP。
   - 教練端：先用教練帳號或手機測試。
3. 決定儲值流程第一版是否採用「會員送出儲值需求，管理端確認後才加票券」。
4. 確認 Supabase 要先沿用 `app_state` 過渡，或直接開始改成 `supabase-frontend-schema.sql` 的正式資料表。
5. 若要正式多端同步，下一步優先做：
   - 會員資料正式表
   - 預約正式表
   - 票券正式表
   - 簽到正式表
   - 權限規則
