# 目前狀態與下一步

更新日期：2026/05/25

這份文件是給 Codex、你自己、或之後協助開發的人快速接上進度用。每次開始修改前，先看這份即可。

## 目前專案定位

- 這是有肌訓練的健身房後台預約管理系統原型。
- 後台目前以電腦版為主。
- 員工手機端與會員端之後另外製作，但要共用同一份資料。
- 目前主要是展示版與本機測試版，不應上傳真實會員資料。

## 目前技術狀態

- 前端：`index.html`、`css/app.css`、`js/app.js`。
- 本機開啟：點兩下 `開啟預約系統.bat`，再打開 `http://127.0.0.1:8765/index.html`。
- GitHub：用來同步程式碼，方便家裡與公司接續開發。
- Supabase：已準備初版 schema，但正式資料同步尚未完成。
- 資料儲存：目前仍主要使用瀏覽器本機資料。

## 重要提醒

- GitHub 同步的是程式碼，不是瀏覽器本機資料。
- Chrome、Codex 內建瀏覽器、公司電腦、家裡電腦的本機資料可能不同。
- 真實會員 Excel、CSV、電話、生日、付款紀錄不能上傳 GitHub。
- `.gitignore` 已排除 Excel、CSV 與 `PRIVATE_DO_NOT_UPLOAD_*`。

## 已完成的核心功能

- 預約行事曆。
- 30 分鐘刻度顯示。
- 60 分鐘課程小卡。
- 過去時段不可新增預約。
- 同時段最多 3 組一般預約。
- 小班團課獨立容量。
- 同一教練同一時段不可重複預約。
- 預約小卡可拖曳移動。
- 拖曳錯誤用右下角訊息提示。
- 預約小卡可顯示簽到章、續約章、分期章。
- 新增預約改成兩步驟：先選課程與教練，再填細節。
- 非離峰時段不顯示友善教練課。
- 自主訓練新增預約時不需要填教練，教練欄位會自動停用。
- 小班團課可先建立空班，也可一次加入多位學員；團體課票券不得分期。
- 會員管理、儲值、票券、課程管理、教練管理、營運摘要已有原型。

- 已增加多位有團課票券的測試會員，方便測試團課陸續加入學員的流程。

  - 團課。
  - 體驗課。
  - 教練課、友善教練課、自主訓練。

## 最近確認過的營運規則

- 分期提醒優先於續約提醒。
- 分期代表原合約尚未完成付款，不需要重新簽約。
- 續約代表同類票券全部用完，需要重新簽約。
- 同一張預約小卡不應同時顯示分期與續約。
- 友善教練課只在平日 09:00 到 18:00 可預約。
- 教練名單目前為：
  - Randy：正職教練。
  - Sandy：店長。
  - Mango：正職教練。
  - Barry：兼職教練。
  - Zoe：合作教練。
  - 小曾：行政櫃台。

## 目前優先順序

1. 穩定預約行事曆操作。
2. 穩定會員與票券資料。
3. 完成 Supabase 資料同步，解決家裡、公司、員工端資料不同步問題。
4. 再製作員工手機端。
5. 再製作會員端。
6. 最後加入登入權限、合約、退款、發票與正式報表。

## 接下來建議先做

### 第一優先：資料同步

目標：家裡、公司、外部 Chrome、未來員工端都讀同一份資料。

需要：

- Supabase Project URL。
- Supabase anon public key。
- 在 Supabase 執行 `supabase-schema.sql`。
- 將 `js/app.js` 的展示模式切換成 Supabase 模式。

### 第二優先：保護真實資料

目標：可以安心給朋友看、上傳 GitHub、放 Google 雲端。

需要：

- 保留展示假資料。
- 真實會員資料只放 Supabase 或私有檔案。
- 不把 Excel 或 CSV 上傳 GitHub。

### 第三優先：介面穩定

目標：櫃台每天操作不卡、不卡視線。

需要：

- 持續測試新增預約、拖曳、簽到、儲值、退費。
- 發現錯誤就用右下角訊息提示。
- 需要確認的修改才跳確認視窗。

## 開發時請避免

- 不要再把大量真實會員資料直接寫進 `js/app.js`。
- 不要把 Excel 轉成程式碼塞進專案。
- 不要一次大改整個會員管理與行事曆。
- 不要讓測試資料與正式資料混在一起。

## 2026-05-25 最新調整

- 團體課簽到不再儲值自主訓練點數，只記錄出席。
