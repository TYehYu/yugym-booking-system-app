# 專案保養機制

這份文件是為了讓 Codex 每次接續開發時保持順暢，不用重新翻整串對話。

## 目標

- 減少重複理解需求的時間。
- 避免已討論過的規則被忘記。
- 讓家裡、公司、未來協作者都能快速接上。
- 把長對話整理成短文件，讓開發速度維持穩定。

## 每次開工前

Codex 開始修改前，請先讀這些文件：

1. `docs/CURRENT_STATUS.md`
2. `docs/NEXT_ACTIONS.md`
3. `docs/SYSTEM_RULES.md`
4. `docs/DECISION_LOG.md`

如果任務跟資料同步、公司電腦或 Supabase 有關，再讀：

5. `docs/WORK_FROM_COMPANY.md`
6. `SUPABASE_SETUP.md`

## 每次完成一輪修改後

只要完成一個明確段落，例如預約功能、會員管理、課程管理、資料同步，就要更新：

- `docs/CURRENT_STATUS.md`：目前做到哪裡。
- `docs/NEXT_ACTIONS.md`：下一步要做什麼。
- `docs/DECISION_LOG.md`：有沒有新增重要決策。

如果只是小修，例如文字、顏色、按鈕位置，可以不用每次更新所有文件，但至少在下一次整理時補上。

## 每次需求變更時

如果你說了新的營運規則，請放進：

- `docs/SYSTEM_RULES.md`

例如：

- 分期優先於續約。
- 同教練同時段不可重複預約。
- 友善課只能平日離峰。
- 小班團課容量規則。

## 每次遇到問題時

如果某個問題反覆發生，請放進：

- `docs/KNOWN_ISSUES.md`

格式：

```text
問題：
影響：
目前處理方式：
之後建議：
```

## 每週或每個大版本整理一次

建議每完成一大段，就請 Codex 做一次：

```text
幫我保養專案文件
```

Codex 應該做：

1. 檢查 `docs/CURRENT_STATUS.md` 是否過期。
2. 檢查 `docs/NEXT_ACTIONS.md` 是否還符合現在目標。
3. 把新規則補進 `docs/SYSTEM_RULES.md`。
4. 把重要決策補進 `docs/DECISION_LOG.md`。
5. 把已解決問題從 `docs/KNOWN_ISSUES.md` 移除或標記完成。

## 建議你之後可以直接這樣說

### 開始新一輪工作

```text
先讀保養文件，接著幫我處理今天的需求
```

### 結束一輪工作

```text
幫我更新目前狀態跟下一步
```

### 感覺 Codex 開始慢或混亂

```text
幫我保養專案文件，整理目前狀態
```

## 保養原則

- 文件要短，不要寫成長篇報告。
- 只記會影響未來開發與營運判斷的內容。
- 已經不重要的細節要刪掉。
- 真實會員資料不要寫進文件。
- 技術細節要用白話說明，讓老闆、店長、協作者都看得懂。
