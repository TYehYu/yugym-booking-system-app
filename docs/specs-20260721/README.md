## 落差分析（2026-07-21，接手時的系統現況）

### 1. 票券「remaining / reserved」模型 ≠ 現行扣課模型 🔴 核心落差
文件假設票券有 `remaining`（剩餘）與 `reserved`（預約保留）兩個數字，預約時 reserved+1、
上課後才真正扣。**現行系統是「預約當下即扣課（deduct）、取消退回（refund）」**，
`member_tickets` 只有 `sessions_remaining`，無 reserved 欄位。
→ 兩條路：①維持現行模型，把文件規則轉譯（「remaining=0 且 reserved=0」＝「剩餘 0 且無未來預約」）；
②真的改成 reserved 模型（動 fn_create/cancel/checkin RPC＋帳本，工程大、風險高）。
**建議①，待使用者確認。**

### 2. 用完票券移「歷史區」、過期票券可「重新啟用」
前端顯示層可做（會員票券列表分區：可用／已用完(歷史)／已過期），無需新表——
文件的 `ticket_history` / `expired_tickets` 建議理解為**顯示分區**而非實體資料表
（現行 member_tickets 已有 status/expire_date 足以判別）。
「重新啟用」語意待定義：是延長 `expire_date`？還是改 status？**需使用者決策**（含權限：僅管理員？）。

### 3. 教練請假流程（代課選單加「請假」）
現行無 `coach_leave` 狀態。文件流程＝退票（+1）＋ booking 保留 status=coach_leave ＋
會員到場簽到仍發 +2 點。工程面：
- 前端：代課名單加「請假」項；行事曆/首頁課卡的 coach_leave 顯示樣式。
- 後端：`fn_cancel_booking`/`fn_checkin_booking` 與 `handle_checkin_reward` 都要認得
  coach_leave（保留 booking 但退票；簽到時不再扣票但發點）——**需新 migration，先在測試庫驗**。

### 4. 場地租借 Facility Rental（$200/次）
新預約類型：不扣票、不發點、不列教練堂數、列「場租收入」。
工程面：新增課程類別/booking type、新增預約流程分支、營收統計（今日營收 KPI 要不要含場租？）、
行事曆卡樣式。**中等工程，前端為主＋營收歸類決策。**

### 5. 體驗課改「先建正式會員 → Trial Booking」
現行體驗課用 `trial_name/trial_phone` 暫存（不建會員）。改為先建會員再約，
可沿用「後台辦理會員」流程串接。**前端流程改造，無 DB 變更。**

### 6. 家庭共用帳號
政策宣告：同帳號可供家庭成員使用、匯入不拆分。系統面今天已支援
「團體課同名重複報名」（帶長輩），其餘為營運政策、無需工程。

### 7. 匯入資料架構（*_import 十表）
文件列 members/coaches/benefits/bookings/checkins/spaces/waiting_list 等匯入表。
注意：**現行系統沒有 spaces（場地資料表）與 waiting_list（候補）功能**（候補在 Dashboard V2
時代已確認全系統 0 處）。此章比較像「舊系統資料重新清整再匯入」的計畫。
**需使用者說明**：匯入來源（哪個舊系統/檔案）、時機（正式庫已有 IMP-* 匯入資料，是增量還是重灌？）、
以及 spaces/waiting_list 是否代表要新開這兩個功能。
