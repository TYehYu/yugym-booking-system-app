# 前台串接規格與交接內容

更新日期：2026/05/25

這份文件是給「管理端、會員端、教練端」共用的串接規格。目標是讓老闆後台、會員手機端、教練手機端之後都接同一份資料，不會各自存一份造成預約、票券、簽到不同步。

## 一句話目標

管理端負責排課、會員、票券、營運管理；會員端負責看自己的預約、簽到、儲值；教練端負責看自己的課表、打卡、幫會員紀錄訓練。

## 目前狀態

目前展示版主要是前端頁面，資料多半存在瀏覽器或整包同步用的 `app_state` 裡。

這種方式適合展示和快速測試，但不適合正式讓會員端、教練端一起操作，原因是：

- 權限不好切：會員端不應該看到全部會員資料。
- 資料不好查：例如只查某位教練今天課表、某位會員票券，整包資料會越來越慢。
- 操作容易互相覆蓋：兩台電腦同時修改時，整包同步比較容易互蓋。

因此串接分兩階段：

1. 展示對齊階段：三邊先使用同一套欄位名稱與規則。
2. 正式資料庫階段：把資料拆成正式表格，並設定會員、教練、管理端權限。

## 三個端的分工

### 管理端

目前這個後台就是管理端。

主要功能：

- 會員管理
- 票券儲值與調整
- 預約行事曆
- 課程管理
- 教練管理
- 營運摘要
- 修改紀錄查詢

管理端可以看到全部資料，也可以做新增、修改、取消、票券調整。

### 會員端

會員手機端建議第一版做這些功能：

- 登入：先用手機號碼登入，之後可接 LINE 或 OTP。
- 我的預約：顯示未來預約與今日課程。
- 簽到：只顯示可以簽到的課程。
- 我的票券：顯示有效票券、剩餘堂數、到期日。
- 儲值：送出儲值需求或付款紀錄，第一版建議先由後台確認後才正式加票券。
- 個人資料：姓名、電話、生日、LINE ID。

會員端只能看自己的資料，不能看到其他會員、其他教練完整資料。

### 教練端

教練手機端建議第一版做這些功能：

- 登入：先用教練帳號或手機登入。
- 今日課表：只顯示自己的課。
- 課程詳情：會員姓名、課程類型、備註、是否簽到。
- 打卡：正職、櫃台、店長需要打卡；兼職與合作教練可先不強制。
- 訓練紀錄：幫會員紀錄今日訓練內容、重量、組數、備註。

教練端不應該看到完整營收、完整會員名單、全部票券金額。

## 共用資料模型

以下是三邊應該一起使用的資料概念。名稱可以照實作調整，但意思要一致。

### members 會員

用途：會員基本資料。

建議欄位：

- `id`：會員 ID
- `name`：姓名
- `phone`：電話，需避免重複
- `line_id`：LINE ID
- `gender`：性別
- `birthday`：生日
- `level`：等級，新朋友、會員、主顧客
- `registered_at`：註冊日期
- `last_active_at`：最近上課或預約日期
- `wallet_amount`：會員儲值金額
- `created_at`
- `updated_at`

等級規則：

- 新朋友：尚未完成教練課方案消費。
- 會員：曾經完成教練課方案消費。
- 主顧客：店內手動標記或之後依消費、上課堂數升級。

### staff_members 員工 / 教練

用途：教練、店長、櫃台與行政人員。

建議欄位：

- `id`
- `name`
- `role`：店長、正職教練、兼職教練、合作教練、行政櫃台
- `level`：例如 LV1、LV2
- `phone`
- `status`：在職、停用
- `clock_required`：是否需要打卡
- `created_at`
- `updated_at`

目前名單：

- Randy：正職教練
- Sandy：店長
- Mango：正職教練
- Barry：兼職教練
- Zoe：合作教練
- 小曾：行政櫃台

### courses 課程

用途：可被預約的課程。

建議欄位：

- `id`
- `name`：教練課1V1、教練課1V2、友善教練課1V1、友善教練課1V2、自主訓練、小班團體課、體驗課
- `type`：coach、friendly、group、self、trial
- `ticket_type`：對應可使用的票券類型
- `duration_minutes`：預設 60
- `capacity_rule`
- `time_rule`
- `color`
- `is_active`

重要規則：

- 教練課、友善教練課、體驗課一般為 60 分鐘。
- 行事曆以 30 分鐘為單位。
- 友善教練課只能平日離峰 09:00-18:00。
- 自主訓練不需要填教練。
- 同一位教練同時段不能重複預約。

### ticket_buckets 會員票券

用途：會員實際擁有的票券，每次儲值、分期、贈送都應該分開存。

建議欄位：

- `id`
- `member_id`
- `ticket_type`
- `label`：教練1V1、教練1V2、友善1V1、友善1V2、團體課、自主、友善自主、贈送教練課
- `source`：purchase、installment、bonus、adjustment、checkin_reward
- `total_count`
- `used_count`
- `remaining_count`
- `expires_at`
- `price_per_class`
- `is_bonus`
- `installment_group_id`
- `installment_index`
- `installment_total`
- `created_at`
- `updated_at`

票券規則：

- 分期票券要分開顯示、分開計算。
- 贈送堂數要另外建立票券，不要混進原本購買堂數。
- 使用完的票券預設不顯示在有效票券，但要保留紀錄。
- 團體課不得分期。
- 自主訓練有到期日。
- 友善自主也有到期日，且只能離峰使用。

### ticket_adjustments 票券異動紀錄

用途：所有手動修改票券都要留下紀錄。

建議欄位：

- `id`
- `ticket_bucket_id`
- `member_id`
- `adjustment_type`：系統調整、延長效期、補發票券、扣除錯誤修正、手動新增、退堂轉儲值金
- `before_snapshot`
- `after_snapshot`
- `reason`
- `operator_id`
- `created_at`

### bookings 預約

用途：行事曆上的預約小卡。

建議欄位：

- `id`
- `course_id`
- `course_type`
- `date`
- `start_time`
- `end_time`
- `coach_id`
- `status`：booked、cancelled、completed
- `note`
- `series_id`：固定預約 ID
- `series_slot_number`：一週一次或一週兩次時，用來判定是哪一天的固定課
- `created_at`
- `updated_at`

重要規則：

- 同一時段最多 3 組一般預約。
- 團課同一時段只能一組，但可以有 1-5 位學員。
- 團課可以先建立空班，再陸續加入學員。
- 同一位教練同時段不能重複預約。
- 過期時段不可新增預約。
- 取消固定預約或移動固定預約時，要詢問是否只改單堂或一併變更後續課程。

### booking_members 預約學員

用途：處理團課多人、1V2 或未來更多人數情境。

建議欄位：

- `id`
- `booking_id`
- `member_id`
- `ticket_bucket_id`
- `class_number`
- `class_total`
- `checked_in_at`
- `checkin_reward_given`
- `created_at`

規則：

- 1V2 雖然有兩人上課，但實務上可能只有一人使用票券，需由 `ticket_bucket_id` 指定扣哪個會員票券。
- 團課每位學員各自扣自己的團課票券。

### payments / recharge_records 儲值與付款

用途：紀錄會員儲值、續約、分期付款、付款方式。

建議欄位：

- `id`
- `member_id`
- `plan_name`
- `ticket_type`
- `payment_method`：現金、匯款
- `amount_due`
- `wallet_used`
- `amount_paid`
- `installment_mode`：none、split_2、split_3
- `installment_index`
- `installment_total`
- `status`：pending、paid、cancelled
- `created_at`

規則：

- 點匯款時，會員端可以顯示匯款資訊。
- 分期只允許系統給定的堂數選項，不讓使用者自行輸入開放堂數。
- 分期最後一期才補贈送堂數。
- 分期優先於續約提醒，因為分期代表合約還沒完成所有付款，不需要重新簽約。

### checkins 簽到

用途：記錄簽到事件。

建議欄位：

- `id`
- `booking_id`
- `member_id`
- `checked_in_at`
- `source_app`：admin、member、coach
- `reward_ticket_bucket_id`
- `created_at`

簽到規則：

- 每堂課每位會員只能簽到一次。
- 只能在課程開始前 60 分鐘內簽到。
- 例：09:00 的課，最早 08:00 簽到。
- 當天簽到教練課，才自動儲值自主訓練 2 點，有效期 1 週。
- 當天簽到友善教練課，才自動儲值友善自主 2 點，有效期 1 週，限平日離峰使用。
- 團體課簽到只記錄出席，不儲值自主訓練。

### staff_clock_records 員工打卡

用途：教練端或員工端打卡。

建議欄位：

- `id`
- `staff_id`
- `clock_in_at`
- `clock_out_at`
- `source_app`
- `note`

### training_records 訓練紀錄

用途：教練幫會員記錄訓練內容。

建議欄位：

- `id`
- `booking_id`
- `member_id`
- `coach_id`
- `training_date`
- `summary`
- `items`
- `note`
- `created_at`
- `updated_at`

第一版 `items` 可以先用文字或 JSON：

```json
[
  {
    "name": "深蹲",
    "weight": "40kg",
    "sets": "4",
    "reps": "8",
    "note": "動作品質穩定"
  }
]
```

## 權限原則

### 管理端

可以：

- 讀取全部資料
- 新增、修改、刪除預約
- 修改會員資料
- 調整票券
- 查看營運摘要
- 查看修改紀錄

### 會員端

可以：

- 讀取自己的會員資料
- 讀取自己的預約
- 讀取自己的票券
- 送出自己的簽到
- 送出自己的儲值需求

不可以：

- 看到其他會員資料
- 看到營運摘要
- 直接修改票券堂數
- 看到完整教練薪資或業績

### 教練端

可以：

- 讀取自己的課表
- 讀取自己課程中需要的會員姓名與備註
- 幫自己的課程寫訓練紀錄
- 打卡

不可以：

- 看全部會員名單
- 看全部營收
- 修改會員票券
- 修改其他教練課表

## 第一版串接建議

### 先不要讓會員端直接改票券

會員端儲值第一版建議只建立「儲值申請」：

1. 會員選方案。
2. 會員選現金或匯款。
3. 如果匯款，顯示匯款資訊。
4. 系統建立一筆 `payments` 或 `recharge_requests`。
5. 管理端確認收款後，再正式建立票券。

這樣比較不會發生「會員付款還沒確認，票券已經增加」的問題。

### 教練端先做今日課表與訓練紀錄

教練端第一版最有價值的是：

- 今天要上誰的課
- 這位會員上到第幾堂
- 上次練什麼
- 今天訓練紀錄

薪資、堂數統計可以先由管理端算。

### 簽到要由系統統一判定

不管是管理端、會員端、教練端，只要做簽到，都必須呼叫同一套規則：

- 是否是同一堂課
- 是否已簽到過
- 是否在可簽到時間內
- 是否需要發自主訓練點數
- 發哪一種自主訓練點數

不要讓三個端各自寫一套簽到邏輯。

## 給太太的對話內容

下面這段可以直接複製貼給太太：

```text
我們後台目前是「管理端」，之後希望你做的會員端跟教練端，可以跟管理端共用同一份 Supabase 資料。

先不用急著把所有功能都做滿，第一版希望先做這些：

會員端：
1. 用手機號碼登入或先做測試登入。
2. 顯示自己的預約。
3. 顯示自己的票券與剩餘堂數。
4. 可以簽到，但簽到規則要跟後台一致：
   - 每堂課只能簽到一次。
   - 課程開始前 60 分鐘內才能簽到。
   - 團課簽到不會送自主訓練。
   - 教練課當天簽到才送自主訓練 2 點，有效 1 週。
   - 友善教練課當天簽到才送友善自主 2 點，有效 1 週，而且只能平日離峰用。
5. 可以送出儲值需求，但第一版先不要直接增加票券，等後台確認收款後再加。

教練端：
1. 教練登入後，只看到自己的今日課表。
2. 可以查看課程會員、時間、備註。
3. 可以打卡。
4. 可以幫會員新增訓練紀錄，例如訓練項目、重量、組數、備註。

資料名稱請盡量跟這份規格一致：
- members：會員
- staff_members：教練與員工
- courses：課程
- ticket_buckets：會員票券
- bookings：預約
- booking_members：預約中的會員
- payments / recharge_records：儲值與付款
- checkins：簽到
- training_records：訓練紀錄

權限上要注意：
- 會員端只能看自己的資料。
- 教練端只能看自己的課表和上課需要的會員資料。
- 票券調整、營收、薪資、全部會員名單，都只放在管理端。

後續我們要把管理端、會員端、教練端都接到同一個 Supabase，這樣公司電腦、家裡電腦、手機端看到的資料才會一致。
```

## 接下來需要一起確認

1. 會員端登入方式：先用手機號碼測試，還是直接接 LINE / OTP。
2. 儲值是否要先做「送出申請，後台確認」。
3. 教練端能不能看到會員電話：建議第一版不要，除非營運真的需要。
4. 訓練紀錄第一版要用純文字，還是表格式項目。
5. Supabase 要先用整包 `app_state` 過渡，還是直接拆正式表格。

## 建議開發順序

1. 先整理正式資料表。
2. 管理端先改成讀寫正式資料表。
3. 會員端先做登入、我的預約、我的票券。
4. 會員端加入簽到。
5. 教練端加入今日課表與打卡。
6. 教練端加入訓練紀錄。
7. 最後再做付款通知、匯款資訊、正式權限與紀錄查詢。
