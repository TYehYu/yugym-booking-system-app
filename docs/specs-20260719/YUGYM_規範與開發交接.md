# YUGYM 預約系統｜規範與開發交接

> 日期：2026-07-19  
> 目的：提供 Claude 接手更新現有 `TYehYu/yugym-booking-system-app`。  
> 原則：延續現有系統，不重做；先完成內部測試 MVP，再逐步上正式環境。  
> 安全：GitHub repository 為公開，不可提交密碼、service role key、正式會員個資或測試帳密。

## 1. Claude 接手指令

請先閱讀本文件與同包其他規格，再檢查目前 branch 的實作狀態。所有新功能必須：

1. 先套用於 `yugym-sprint3-test`，不得直接修改正式 Supabase。
2. 保留既有預約、票券、會員與員工資料相容性。
3. 權限必須同時由前端導覽與 Supabase RLS／RPC 控制；不可只隱藏按鈕。
4. 管理員、櫃台、教練、主管、會員均需使用實際帳號驗收，管理員「預覽」不算權限測試。
5. schema 變更必須留下 migration、回滾／切換說明及 UAT 項目。
6. 不得在公開 repo 保存任何密碼。

## 2. 角色與工作區規範

### 櫃台

櫃台登入後只看到每天必要的四個主要區域：

- 今日櫃台
- 預約
- 會員
- 票券・收款

首頁快速操作必須能直接進入：新增預約、新增會員、售票／儲值、今日會員簽到管理。

注意：「會員課程簽到」與「員工上下班打卡」是兩個不同流程，不得混用同一入口。

### 管理員

- 導覽列顯示「管理員」。
- 可進入人員、規則、票券設定、系統設定及營運管理功能。
- 管理員功能與櫃台日常操作區分開，避免櫃台看到不需要的設定。

### 主管

- 目前唯一職責：排班。
- 導覽列顯示「主管」，主管區目前只放排班。
- 主管統一月初上任，不做月中按日比例計算。

### 身兼多職

- 同一帳號若同時是管理員與主管，導覽列要同時顯示「管理員」和「主管」。
- 權限採能力疊加，不能以單一 role 覆蓋另一身分。

### 教練

- 只能查看本人相關課程、學員、出勤與薪酬資料。
- 不能看到其他教練的會員敏感資料或管理員設定。

## 3. 員工類型與薪酬規則

所有規則須採「範本 + 個人覆寫」模型：先依聘僱類型套預設值，管理員可針對單一員工調整，並保留生效起訖日期與異動紀錄。

### 正職

- 保障底薪：29,500 元／月。
- 值班：每週固定 5 天，每天 6 小時。
- 教練課／友善課堂獎金採分段累進：
  - 第 1–59 堂：600 元／堂。
  - 第 60–79 堂：650 元／堂。
  - 第 80–99 堂：675 元／堂。
  - 第 100 堂以上：700 元／堂。
- 達標獎金：
  - 80 堂：5,000 元。
  - 100 堂：追加 3,000 元；達 100 堂合計 8,000 元。
- 團課獎金：200 元／實到人數。
- 有勞健保。

### 兼職

- 時薪：200 元。
- 值班：每週至少 3 天，每天 6 小時。
- 教練課／友善課：600 元／堂。
- 團課：200 元／實到人數。
- 有勞健保。

### 合作

- 可不值班；若值班，時薪 200 元。
- 教練課／友善課：600 元／堂。
- 團課：200 元／實到人數。
- 無勞健保。

### 工讀

- 時薪：200 元。
- 無每週最低值班天數，也可以完全不值班。
- 教練課／友善課：600 元／堂。
- 團課：200 元／實到人數。
- 無勞健保。

### 主管與店長獎金

- 主管獎金：4,000 元／月。
- 店長可同時領主管獎金。
- 店長獎金按「分店、月份、達標教練人數」計算：
  - 某位教練達 80–99 堂：4,000 元。
  - 某位教練達 100 堂以上：再加 2,000 元，該教練共 6,000 元。
  - 兩位教練達標領兩份，依此類推。

### 堂數認列

- 只統計已完成／已簽到的教練課與友善課。
- 體驗課、贈送課、補課不計入教練堂獎。
- 補簽是原課程的狀態修正，不得重複計堂或重複發獎金。
- 團課依實際簽到人數計算，不依預約人數。

## 4. 續約獎金

- 所有教練統一：1,000 元／張。
- 只適用教練課與友善課票券。
- 一張續約票券只能歸一位教練，不拆分 500／500。
- 預設歸會員主教練，但櫃台可在成交時指定其他一位教練。
- 分期付款目前定案採較保守方案：第一期付款時先建立待認列紀錄，全部款項繳清後才正式認列 1,000 元。
- 票券退款或作廢後必須追回；以負向獎金紀錄保留稽核，不可直接刪除歷史。

## 5. 會員註冊與舊資料連結

### 新會員

- 會員可自行註冊 Auth 帳號並建立自己的 member 資料。
- 未登入／匿名使用者不可讀取會員、票券、預約等營運資料。

### 原會員連結舊資料

- 若手機號碼對應既有會員，不可直接覆寫原會員 `auth_id`。
- 建立 `member_link_requests` 待審申請。
- 櫃台／管理員核對姓名、手機及館內紀錄後才可核准或拒絕。
- 核准後保留原 member ID、票券、預約、紀錄及主教練關係。
- 必須防止帳號搶綁與重複會員資料。

## 6. 登入與 LINE 方向

- 員工／管理後台維持帳號＋密碼登入。
- 會員未來由 LINE 官方帳號／LINE Login 進入系統；功能完成前保留手機＋密碼登入。
- LINE 入口必須以 feature flag 控制；未設定 Channel ID、callback URL 等必要資訊時不可顯示失效按鈕。
- LINE 身分仍須安全連結到 member，不可只用可修改的前端資料作授權。

## 7. 票券規則與彈性

票券不能寫死成少數方案，必須支援：

- 票券類型、名稱、堂數、效期、生效方式。
- 可用課程類別：教練課、友善課、團課、自主訓練等。
- 預約扣票、取消退票、24 小時內取消、補課券與補簽規則。
- 一次付款或分期付款、各期開通堂數、待收款狀態。
- 暫停、作廢、退款、轉移等狀態與稽核紀錄。
- 規則範本可供多張票券共用；個別票券可在授權範圍內覆寫。
- 規則變更需有生效日期；既有已售票券不得被新規則無聲改寫。

## 8. 場地規則與彈性

- 場地需能設定類型、容量、可用時段及適用課程。
- 預約時要檢查教練、會員、場地時間衝突。
- 不同分店可有不同場地與規則。
- 管理端可調整場地設定，但櫃台日常畫面只需看到可預約／衝突結果。

## 9. 介面方向

- 視覺：YUGYM 品牌綠工作畫布、白色大圓角資訊卡、深綠操作卡、少量金色提醒。
- 桌機：左側角色導覽；上方登入身分／日期／時間；中央今日營運；右側待辦與快速操作。
- 首頁不是報表中心，只保留每日要採取行動的資訊。
- 異常與待辦要突出，正常狀態保持安靜。
- 管理員與櫃台共用同一套版面元件，以權限過濾內容；避免維護兩套 UI。
- 手機版資訊不可消失，可改為換行、摺疊或捲動。

## 10. 測試環境與安全界線

- 正式 Supabase：`yugym-booking-system`，不得在未明確批准下修改。
- 測試 Supabase：`yugym-sprint3-test`。
- 本機測試網址使用 `?env=test`，畫面右下必須顯示 `TEST 測試環境`。
- 測試帳密不可提交到 repo 或打包進 ZIP；Auth 密碼只能重設，不能讀回。
- 生產部署前必須完成管理員、櫃台、教練、會員與匿名越權測試。

## 12. 下一階段優先順序

1. 建立聘僱範本、員工個別覆寫與生效日期模型。

3. 實作主管、店長、團課及續約獎金 ledger。
4. 產生月結試算、明細、人工調整、鎖定與重算流程。

### P2：會員入口與營運完善

1. LINE 官方帳號／LINE Login。
2. 票券規則範本與個別覆寫管理介面。
3. 場地衝突與分店規則。
4. 手機版與無障礙／操作效率優化。

## 13. 尚需 Claude 特別檢查

- 前端某些舊註解／文件可能仍寫「教練可簽到自己的課」，但目前產品決策是會員自行掃碼、櫃台可代簽；請統一程式、UAT 與文件。
- 後台新增會員畫面仍可能出現舊的預設密碼敘述；應與新的安全註冊／舊會員連結流程統一，避免櫃台誤以為系統會發放共用密碼。
- 任何 `SECURITY DEFINER` RPC 都要明確檢查 `auth.uid()` 與角色，撤銷 `PUBLIC` execute，再只授權必要角色。
- RLS 不可只使用 `TO authenticated`；必須加所有權或職務判斷。
- 管理員預覽櫃台畫面只能測 UI，不能作為 RLS 驗收結果。

## 14. 完成定義

內部 MVP 可交付的最低標準：

- 櫃台可登入並完成新增會員、預約、售票／收款、簽到與取消。
- 教練可登入並查看本人課表、學員、出勤與薪酬試算，不可查看他人敏感資料。
- 管理員可管理必要設定與審核舊會員連結。
- 主管可排班，且身兼角色正確疊加。
- 測試環境無 P0 資安警告、無正式資料異動。
- UAT、migration、回滾與切換文件齊全後，才可規劃正式上線。
