# 重要決策紀錄

這份文件記錄已經討論過、會影響系統設計的決策。

## 2026/05/25

### 後台以電腦版為主

決策：

- 有肌訓練後台系統主要給老闆、店長、櫃台在電腦使用。
- 教練或會員手機版之後另外製作。

原因：

- 後台資料密度高，電腦版比較適合管理預約、會員、票券與營運摘要。
- 手機端需要更簡化，不適合直接塞完整後台。

### GitHub 同步程式碼，Supabase 同步資料

決策：

- GitHub 用來同步程式碼。
- Supabase 用來同步會員、預約、票券與營運資料。

原因：

- GitHub 不適合放真實會員資料。
- 瀏覽器本機資料無法可靠跨電腦同步。

### 展示版保留假資料

決策：

- 展示版使用假會員與假預約。
- 真實會員資料不得寫進 `js/app.js` 或上傳 GitHub。

原因：

- 保護會員隱私。
- 方便給朋友、工程師或外部人員看系統，不暴露真實資料。

### 分期提醒優先於續約提醒

決策：

- 同一張預約小卡不應同時顯示分期與續約。
- 如果該會員還有下一期付款未開放堂數，優先顯示分期。
- 只有同類票券全部用完且沒有後續分期時，才顯示續約。

原因：

- 分期代表同一份合約還沒走完，不需要重新簽約。
- 續約代表要重新簽一份合約。

### 友善教練課限制離峰

決策：

- 友善教練課只在平日 09:00 到 18:00 可預約。
- 新增預約時，非離峰時段不顯示友善教練課。

原因：

- 避免櫃台誤排。
- 符合實際營運規則。

### 同教練同時段不可重複預約

決策：

- 同一個教練同一時段不能有兩張教練課預約。

原因：

- 防止實務排課衝突。

### 小班團課獨立容量

決策：

- 一般預約同時段最多 3 組。
- 小班團課同時段只能有一組。
- 團課不與一般預約容量互斥。

原因：

- 團課與一般預約的管理邏輯不同。
### 2026-05-25 簽到點數區分

- 團體課簽到不發自主訓練點數，避免團課出席造成額外自主訓練儲值。
- 友善教練課簽到發「友善自主」，用途限定平日離峰時段，與一般自主訓練分開計算。
- 預約小卡儲存若沒有修改內容，直接關閉；只有有改動時才要求確認。
## 2026/05/25 前台與教練端串接方向

補充：
- 使用者準備休息，串接先暫停。
- 之後使用者只要輸入「串接」，就從 `docs/FRONTEND_INTEGRATION.md` 與 `supabase-frontend-schema.sql` 開始執行正式串接流程。

決定：
- 管理端、會員端、教練端未來共用同一份 Supabase 資料。
- 會員端第一版只處理自己的預約、票券、簽到、儲值需求。
- 教練端第一版只處理自己的課表、打卡、訓練紀錄。
- 會員端儲值第一版建議先建立儲值需求，由管理端確認收款後才正式增加票券。
- 正式資料庫應拆成會員、員工、課程、票券、預約、簽到、付款、訓練紀錄等資料表，不再長期依賴整包 `app_state`。

原因：
- 這樣才能保護會員資料，避免會員端看到全部會員。
- 教練端只需要看到自己上課需要的資訊，不需要看到營收與完整會員名單。
- 票券、簽到、付款牽涉到金額與權益，應該由管理端保留最後確認權。

## 2026/05/26

### 後台列表介面統一

決策：

- 會員列表、課程及票券、教練管理都往同一種後台列表規格靠攏。
- 左側放新增、篩選、快速選擇等操作。
- 右側主要顯示列表。
- 詳細資料不常駐在右側，而是點選後用彈跳視窗展示。

原因：

- 櫃台與老闆更容易掃讀。
- 避免畫面資訊過多。
- 未來會員、課程、票券、教練都能用一致的操作習慣。

### 課程管理改名為課程及票券

決策：

- 左側主選單改為「課程及票券」。
- 子項目分成「課程」與「票券」。
- 點父層可收合或展開子項目。

原因：

- 課程與票券是不同資料，但實務上常一起維護。
- 放在同一群組可以降低選單複雜度。
