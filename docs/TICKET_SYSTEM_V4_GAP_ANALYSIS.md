> 建立：2026-07-18。對照 `docs/TICKET_SYSTEM_V4_SPEC.md`（規格）與現行系統（正式庫 `rlpiomzplckzqnqrvrwc` + `index.html`）。
> 方法：正式庫唯讀查詢 schema／資料 + 前端程式碼查找。**未改動任何正式庫資料或程式。**

## 結論（一句話）

**V4 架構的骨架其實已大致實作**——方案/票券分離、快照語意、方案 CRUD、分期前端、自主訓練發點都已存在。
剩下的是**四個明確落差**：方案「複製」與「封存」缺、分期規則與規格不符、自主訓練效期起算點、以及既有的 RPC 到期檢查。
不是從零打造，是「補齊 + 對規格校正」。

## 二、落差（要做的事）

### 落差 1 — 方案「複製」缺（規格 §5、§7）
規格：「所有方案可新增、修改、**複製**」「活動方案以**新增方案**完成，不修改程式」。
現況：方案列表列只有「編輯／下架／刪除」（16437-16439），**全檔無 clone/duplicate**。
影響：辦活動要靠「新增」從頭填，複製一個既有方案改價格會快很多。屬體驗改善、低風險。

### 落差 2 — 方案「封存 archive」缺，狀態只有二態（規格 §5、§6）
規格：狀態三態＝**上架／停用／封存**；刪除規則「已有歷史資料 → **僅能停用或封存**」。
現況：`course_plans` 只有 `active` 布林（上/下架）＋硬刪（`deletePlan` 有守門）。**無 `archived` 欄位、無封存概念**。
差異：「停用」是暫時下架仍可能重新上架；「封存」是永久退役、從一般清單隱藏但保留歷史。目前兩者混在 `active=false`。
影響：需加 `course_plans.archived`（或狀態 enum）＋ UI。中低風險（純加欄位＋前端）。

### 落差 3 — 分期規則與規格不符（規格 §4）
現況：前端分期功能**已完整**（`submitGrant` 17141、`splitSessions` 17092、`openInstallNext`/`confirmInstallNext` 17169/17197、財務待收款統計、會員申請端），但：

| 規格 §4 | 現行實作 | 落差 |
|---|---|---|
| 僅 12 堂／365 天方案可分期 | 無此限制檢查 | 需加驗證 |
| 2 期固定 6+6、3 期固定 4+4+4 | `splitSessions` 平均切 n 段（10 分 3 期→4/3/3） | 演算法不同 |
| 2 期每期 168 天、3 期每期 112 天 | 未見每期天數規則 | 缺 |
| 每期首次使用才開始計時 | 解鎖靠櫃台手動收款（`confirmInstallNext`），非計時 | 語意不同 |

且正式庫 `member_tickets.installment` **實際使用 0 筆** → 功能開發了但沒真的在用。

**✅ 已決策（2026-07-18）：保留現行彈性版，不改規格 §4。** 使用者選擇現行「平均切 n 段 + 櫃台手動收款解鎖」——
可任意堆數、任意期數，比規格固定的 6+6 / 4+4+4 更靈活。故**規格 §4（固定切法、每期 168/112 天、
限 12 堂/365 天、首用計時）視為「已用彈性版取代」，不實作**。現行分期功能維持原狀。

### 落差 4 — 自主訓練效期起算點（規格 §3）
規格：效期固定 7 天、**以課程日為第 1 天**；補簽仍以 `booking.date` 計算效期。
現況：`grantCheckinReward` 用 `expire = 發放日 + 7`（11640）。
差異：當天簽到時兩者相同；但**補簽（隔幾天才簽）時，現行用發放日、規格要用 booking.date**，會多給幾天效期。
影響：小而明確的 bug。改 1 行（起算改 `booking.date`）即可對齊規格。低風險。

### 落差 5（沿用待辦 #6）— RPC 未檢查票券到期日
`fn_create_booking/checkin/cancel` 定義中無 expire 檢查，到期擋在前端。與 7/16 授權漏洞同類（前端擋 ≠ 系統擋）。
永久有效票不受影響；若 V4 恢復有效期票，應在 RPC 補到期檢查。中風險（動核心 RPC，需照 CLAUDE.md 授權清單）。

## 三、非落差（澄清）

- **source_plan_id 正式庫 0 筆**：不是 bug。現行發券流程**有寫** source_plan_id，但正式 2175 筆 `purchase` 皆為**歷史匯入資料**（早於方案系統），56 筆 `checkin_grant` 本就不掛方案。新票會逐步累積。快照正確性靠 `plan_name`/`unit_price` 已達成，`source_plan_id` 只影響「回溯是哪個方案賣的」分析追溯，非功能缺陷。
- **VIP / 限定教練課**：2026-07-18 已結案——確認就是一般教練課（活動價、已停售），`requires_coach` 已補為 true。無資料模型遷移需求。
