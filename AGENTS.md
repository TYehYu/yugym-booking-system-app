# AGENTS.md

## 專案背景

使用者是一位健身房老闆。協助本專案時，請優先考慮健身房經營、會員服務、課程安排、教練管理、場館營運、行銷活動與在地客戶體驗。

## 溝通風格

- 使用清楚、直接、容易執行的中文。
- 避免過度技術化說明；必要時用白話解釋。
- 給建議時要貼近健身房老闆的日常決策，例如招募會員、提高續約率、排課效率、提升客戶滿意度。
- 若需要做選擇，優先推薦低風險、容易落地、能立即改善營運的方案。

## 工作重點

- 會員管理：註冊、續約、分級、出席紀錄、流失預警。
- 課程與教練：團課排程、私人教練預約、教練績效、課程容量。
- 行銷與銷售：試營運活動、體驗課、推薦獎勵、節慶促銷、社群內容。
- 場館營運：設備維護、尖峰時段分流、櫃台流程、客訴處理。
- 財務觀察：月費收入、課程收入、續約率、會員留存、平均客單價。

## 產品與設計偏好

- 介面應該乾淨、實用、容易掃讀，適合櫃台、店長或老闆快速操作。
- 優先呈現今日預約、會員狀態、課程容量、待處理事項與營收摘要。
- 不要做成只有展示效果的頁面；應該能支援真實的健身房工作流程。
- 後台主要以電腦版操作；教練或會員手機版可以另外製作並串接同一份資料。

## 開發與修改原則

- 優先沿用現有專案架構與命名方式。
- 修改範圍保持精準，不做無關重構。
- 新增功能時，先確保核心流程可用，再加上美化或進階功能。
- 若涉及會員資料、付款資料或健康相關資訊，要特別注意隱私、安全與資料正確性。

## 預設目標

協助使用者把專案做成一個能真正改善健身房營運效率、會員體驗與收入管理的工具。

## 固定快捷指令

使用者之後如果只輸入以下短句，請直接照對應流程執行，不需要再反覆確認。

### 保養

代表「幫我整理與優化這個專案，讓後續開發更順」。

執行重點：
- 檢查目前檔案結構與主要文件是否一致。
- 更新 `docs/CURRENT_STATUS.md`、`docs/NEXT_ACTIONS.md`、`docs/SYSTEM_RULES.md`、`docs/DECISION_LOG.md`、`docs/KNOWN_ISSUES.md` 中需要同步的內容。
- 檢查 `index.html`、`css/app.css`、`js/app.js` 是否有明顯重複、壞掉流程、命名混亂或已完成但未記錄的規則。
- 執行可行的語法檢查與瀏覽器基本檢查。
- 回報「目前狀態、已整理內容、下一步建議」。

### 上傳

代表「把目前最新版本同步到 GitHub 展示版」。

執行重點：
- 先檢查不要上傳真實會員資料、Excel、CSV、`PRIVATE_DO_NOT_UPLOAD_*`、log 或其他私密資料。
- 執行可行的語法檢查與基本畫面檢查。
- 將最新變更提交並推送到 GitHub。若命令列工具不可用，改用 GitHub Desktop 流程引導使用者完成 Commit / Push。
- 上傳後提醒使用者等待 GitHub Pages 部署，展示網址為：
  `https://tyehyu.github.io/yugym-booking-system-app/`
